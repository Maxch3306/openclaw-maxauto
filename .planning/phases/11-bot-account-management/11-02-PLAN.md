**AddBotDialog.tsx** -- Modal dialog with step-based flow per user decisions:

Props:
```typescript
interface AddBotDialogProps {
  open: boolean;
  onClose: () => void;
  onAdded: () => void; // callback to reload bot list after successful add
  existingAccountIds: string[]; // to prevent duplicate bot accounts
}
```

**Step-based UI flow (all within one modal, not separate pages):**

1. **Token input step (initial):**
   - Text input for bot token (not password type — user is entering a new token)
   - "Validate" button that triggers getMe API call
   - Validation states: idle, validating (spinner), success (green check + @username), error (red message)
   - Token format regex pre-check: `/^\d+:[A-Za-z0-9_-]{30,}$/`
   - If token belongs to an already-configured bot (check `botUsernameToAccountId(username)` against `existingAccountIds`), show error: "This bot is already configured"
   - Validate button disabled until token format matches regex

2. **Agent selection step (shown ONLY after successful validation — per locked decision):**
   - Show validated bot info: "@username" with green checkmark
   - Agent dropdown: `useChatStore(s => s.agents)` filtered for 1:1 enforcement
   - Same filtering logic as BotCard: check `allBindings` for telegram bindings to other accounts, disable those agents with "(bound to @other_bot)" label
   - Load current bindings via `gateway.request("config.get")` when dialog opens
   - "Save" button disabled until an agent is selected
   - Agent binding is REQUIRED per locked decision

3. **Save flow:**
   - Derive accountId via `botUsernameToAccountId(validatedUsername)`
   - Check `needsMigration(tg)` — if existing flat config with botToken, call `migrateToMultiAccount()` first, then re-fetch config
   - Build new account config: `{ enabled: true, botToken: token.trim(), dmPolicy: "open", groupPolicy: "disabled" }`
   - Build updated bindings via `buildUpdatedBindings(allBindings, accountId, selectedAgentId)`
   - Single `patchConfig()` call with account + bindings
   - `waitForReconnect()`
   - Call `onAdded()` to trigger parent reload
   - Close dialog

4. **Error handling:** Show error message inline if patchConfig fails. Do not close dialog on error.

5. **Cancel/close:** Reset all state (token, validation result, selected agent). Close button (X) in header and Cancel button in footer.

**Styling:** Follow existing modal pattern from CreateAgentDialog — `fixed inset-0 z-50`, `bg-black/50` backdrop, `w-[440px]` dialog (slightly wider for token input), rounded-xl, shadow-2xl. Use project CSS variables throughout.

**RemoveBotDialog.tsx** -- Confirmation modal:

Props:
```typescript
interface RemoveBotDialogProps {
  open: boolean;
  accountId: string;
  username: string; // @username for display
  onClose: () => void;
  onRemoved: () => void; // callback to reload bot list after removal
}
```

**Content:**
1. Warning icon (AlertTriangle from lucide-react) in orange/red
2. Title: "Remove Telegram Bot"
3. Summary of what will be lost:
   - Bot: @{username}
   - Load current config to show: bound agent name (from bindings), DM policy, group policy, number of access list entries
   - Display as a compact info block
4. Warning text: "This will permanently remove the bot configuration and its agent binding. This cannot be undone."
5. Footer: "Cancel" (secondary) + "Remove" (red/destructive styling)

**Remove flow:**
1. Load current config and bindings via `gateway.request("config.get")`
2. Build updated bindings: `buildUpdatedBindings(allBindings, accountId, null)` — removes the binding
3. Single atomic `patchConfig()`:
   ```typescript
   await patchConfig({
     channels: { telegram: { accounts: { [accountId]: null } } }, // merge-patch delete
     bindings: updatedBindings,
   });
   ```
4. `waitForReconnect()`
5. Call `onRemoved()` and close

**Styling:** Destructive dialog pattern — "Remove" button uses `bg-[var(--color-error)]` or `text-[var(--color-error)]` with border styling. Match existing modal pattern.
