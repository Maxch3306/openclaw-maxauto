Create `src/api/telegram-accounts.ts` with all multi-account helper functions. This module is the contract that IMChannelsSection and future Phase 11 UI will use.

**Types to define:**

```typescript
// Extend TelegramConfig to include accounts structure
export interface TelegramAccountConfig {
  enabled?: boolean;
  botToken?: string;
  dmPolicy?: string;
  groupPolicy?: string;
  allowFrom?: Array<string | number>;
  groupAllowFrom?: Array<string | number>;
  groups?: Record<string, Record<string, unknown>>;
}

export interface TelegramConfig extends TelegramAccountConfig {
  accounts?: Record<string, TelegramAccountConfig>;
  defaultAccount?: string;
}

export interface BindingEntry {
  type?: string;
  agentId: string;
  comment?: string;
  match: { channel: string; accountId?: string; [key: string]: unknown };
}
```

**Functions to implement:**

1. `botUsernameToAccountId(username: string): string` - Remove leading `@`, lowercase, replace invalid chars with `-`, strip leading/trailing dashes, max 64 chars. Empty/invalid returns `"default"`. Must match OpenClaw's `canonicalizeAccountId` logic exactly.

2. `isMultiAccountConfig(tg: TelegramConfig): boolean` - Returns true if `tg.accounts` has at least one key.

3. `needsMigration(tg: TelegramConfig): boolean` - Returns true if `tg.botToken` exists at top level AND `accounts` is empty/missing. This means flat single-bot config that needs restructuring when a second bot is added.

4. `getAccountConfigs(tg: TelegramConfig): Map<string, TelegramAccountConfig>` - If multi-account: iterate `tg.accounts`. If legacy flat with `tg.botToken`: return single entry keyed `"default"` with flat fields (destructure out `accounts` and `defaultAccount`). If neither: return empty map.

5. `getTelegramBindingForAccount(bindings: BindingEntry[], accountId: string): BindingEntry | undefined` - Find binding where `match.channel === "telegram"` AND (`match.accountId === accountId` OR (no accountId AND accountId === "default")).

6. `buildUpdatedBindings(allBindings: BindingEntry[], accountId: string, agentId: string | null): BindingEntry[]` - Filter out ONLY this account's telegram binding (matching both channel AND accountId, including legacy no-accountId for "default"). If agentId provided, append new binding with `{ agentId, match: { channel: "telegram", accountId } }`. Return full array.

7. `async migrateToMultiAccount(existingConfig: TelegramConfig, existingBindings: BindingEntry[]): Promise<void>` - Build a single patchConfig call that:
   - Copies ALL flat TelegramAccountConfig fields (botToken, enabled, dmPolicy, groupPolicy, allowFrom, groupAllowFrom, groups) into `channels.telegram.accounts.default`
   - Sets `channels.telegram.defaultAccount` to `"default"`
   - Sets `channels.telegram.botToken` to `null` (merge-patch delete to prevent inheritance to other accounts)
   - Updates any telegram binding without accountId to include `accountId: "default"`
   - Calls `patchConfig(patch)` then `waitForReconnect()`
   - Import patchConfig and waitForReconnect from `./config-helpers`

**Critical details from research:**
- Migration MUST explicitly copy `groups` into accounts.default because OpenClaw's `mergeTelegramAccountConfig` does NOT inherit channel-level groups in multi-account mode
- Top-level `botToken` MUST be set to `null` to delete via merge-patch, otherwise it becomes a fallback for ALL accounts
- Legacy bindings without accountId act as wildcards; migration must add `accountId: "default"` to prevent routing conflicts
- Set `defaultAccount: "default"` during migration to prevent OpenClaw warnings

Replace:
```typescript
const otherBindings = allBindings.filter((b) => b.match?.channel !== "telegram");
const updatedBindings = boundAgentId
  ? [...otherBindings, { agentId: boundAgentId, match: { channel: "telegram" } }]
```

With:

**Refactor `saveTelegramConfig()` config write:**

Replace the flat config write:
```typescript
await patchConfig({
  channels: { telegram: telegramConfig },
  bindings: updatedBindings,
});
```

With account-scoped write:
```typescript
await patchConfig({
  channels: {
    telegram: {
      accounts: {
        [currentAccountId]: telegramConfig,
      },
    },
  },
  bindings: updatedBindings,
});
```

**Note:** For the existing single-bot case where `needsMigration()` would be true, we do NOT trigger migration here. Migration only triggers when adding a SECOND bot (Phase 11). The current single-bot continues to work with flat config until then. The key fix here is the binding filter — it must scope by accountId so Phase 11 can safely add more bots.

**However**, the config WRITE path should always write to the accounts structure going forward. When a user saves settings for their existing single bot, it should write to `accounts.default` (not flat top-level). This is a safe forward-migration — OpenClaw reads both flat and accounts structures. The read path (`loadTelegramConfig`) should handle BOTH shapes via `getAccountConfigs()`.

**Refactor `disableTelegram()`:**
Replace flat write with account-scoped:
```typescript
await patchConfig({
  channels: { telegram: { accounts: { [currentAccountId]: { enabled: false } } } },
```
